**Problem:** `native.ts` has two critical bugs:
1. **Platform resolution:** On Windows, it looks for `spectra_napi.dll` but `require()` on Windows needs `.node` extension for native addons. The DLL IS a valid Node addon (napi-rs produces it), but Node's module loader expects the `.node` extension. After Task 1 fixes `copy-native.js` to always produce `.node` files, the loader should always look for `.node`.
2. **Fallback behavior:** When native fails to load, the fallback `createAgent` returns `JSON.stringify({ error: "native_not_loaded" })` — but `agent.ts` tries to parse this as either a plain agent ID string OR `JSON.parse(result).agent_id`. The `{ error: ... }` format doesn't match either path, causing `agent.ts` to throw `SpectraError("Failed to create agent: {error...}")`. The fallback should match the expected format.

**Fix `packages/spectra-ts/src/native.ts`:**

1. Simplify `getNativeDir()`: Instead of resolving relative to `import.meta.url` (which differs between `src/` and `dist/`), use a two-step lookup:
   - First try: `dist/native/spectra_napi.node` (relative to package root — production)
   - Second try: `src/native/spectra_napi.node` (development with tsx)
   - Use `process.cwd()` or `__dirname` equivalent for resolution

   Actually, the best approach: since `copy-native.js` now copies to BOTH `dist/native/` and `src/native/`, the existing `getNativeDir()` approach works — it resolves relative to the current file's directory. For `dist/native.js`, that points to `dist/../native` which doesn't exist. 

   Better approach: use a single canonical location. Since `copy-native.js` copies to `dist/native/`, and `dist/` is the published output, make `native.ts` always look for `spectra_napi.node` relative to its own directory's sibling `native/` folder. The key fix: **always use `.node` extension**, never `.dll`.

   Replace the platform-dependent extension logic:
   ```typescript
   const ext = process.platform === "win32" ? ".dll" : ".node";
   ```
   With:
   ```typescript
   const nativePath = join(nativeDir, "spectra_napi.node");
   ```
   
   This works because `copy-native.js` now always produces `.node` files regardless of platform.

2. Fix fallback `createAgent` to return a format compatible with `agent.ts`:
   ```typescript
   createAgent: () => JSON.stringify({ error: "native_not_loaded" }),
   ```
   Should be:
   ```typescript
    createAgent: () => { throw new Error("Native addon not loaded. Run 'cargo build --release --package spectra-napi' and 'bun build' to compile."); },
   ```
   Or alternatively, return a format that `agent.ts` can detect:
   ```typescript
    createAgent: () => JSON.stringify({ error: "native_not_loaded", message: "Native addon not loaded. Build with: cargo build --release --package spectra-napi && bun build" }),
   ```
   Since `agent.ts` already handles `{ error }` in the parsed result (line 52-53: `if (parsed.error)`), the issue is that `createAgent` fallback returns a string that when parsed has `{error}` but the code at line 58 checks `if (result && !result.startsWith("{"))` — the fallback DOES start with `{` so it goes to the else branch and throws. This is actually fine behavior — it surfaces the error clearly. But the error message should be more helpful.

   Better fix: make `native.ts` throw a clear error when native isn't loaded, rather than returning a broken string. Update `loadNative()` to track whether native loaded successfully, and have `createAgent`/`runAgentWithInput` throw with a helpful message when native failed:
   ```typescript
   export function isNativeLoaded(): boolean {
     loadNative();
     return native !== null && !isFallback;
   }
   ```

3. Add `isNativeLoaded()` export so consumers can check before calling native methods.
